    Git 版本控制

    Git 与版本控制

    Git：Git 是一个开源免费的分布式版本控制系统

    版本控制：指记录文件内容变化，以便将来查阅特定版本修订情况的系统

    集中式与分布式

    集中式版本控制：

    集中式版本控制

    集中式版本控制有一个对代码集中管理的服务器，贡献者必须连接到该服务器以获取最新文件或提交更新

    • 优点：集中式版本控制可以使每个贡献者在一定程度上看到其他人对项目的操作。管理员也可以掌控每个贡献者的权限
    • 缺点：服务器是集中式版本控制的核心，服务器不能停机

    分布式版本控制：

    分布式版本控制

    分布式版本控制会将代码仓库的镜像完整地下载下来（而不是仅下载最新版本的快照），若代码仓库停机，在修复之后可以使用本地的镜像进行恢复，并且分布式版本控制允许贡献者在本地进行自己的版本控制

    Git 工作机制

    • 工作区：指在本地存放被编写的代码文件，该文件未被

    • 暂存区（临时存储）：指代码交由 Git 在本地暂时托管，相当于目前代码进度的一个备份，你可以将该备份拉回工作区也可以提交

    • 版本库（生成历史版本）：当代码被提交到版本库时，会生成对应的版本号，并且代码的变化被持久化记录

    代码托管服务器

    • GitLab（用于局域网）
    • GitHub（国外）
    • Gitee（国内）

    Git 常用命令

    命令 作用
    git config --global user.name 用户名 设置用户签名
    git config --global user.email 邮箱 设置用户签名
    git init 初始化本地库
    git status 查看本地库状态
    git add 文件名 添加到暂存区
    git restore 文件名 将工作区内容还原为暂存区的内容
    git commit -a （仅对修改和删除的文件有效
    对于新增的文件，则一定要 add）
    直接提交，不需要 add
    git commit -m "日志信息" 文件名 提交到本地库
    git commit --amend -m "日志信息" 修改最近一次提交的日志信息
    git reflog 查看历史记录
    git log 查看详细历史记录
    git reset --hard 版本号 版本穿梭

    细说常用命令——用户签名与本地库

    用户签名

    签名的作用是区分不同操作者的身份。用户签名信息在每一个笨笨的提交信息中都能看到。Git 首次安装必须设置用户签名，否则无法提交代码。并且用户签名与代码托管中心的账号没有任何联系。

    初始化本地库

    进入一个目录，运行 git init 将该目录指定为本地代码仓库

    查看本地库状态

    在已经定义为本地库的目录下，运行 git status，查看本地库状态

    细说常用命令——代码版本管理

    代码追踪（添加到暂存区）

    在定义为本地库的目录下创建任意文件，写入任意内容，保存。当再次查看本地库状态时，git 会显示检测到一个未追踪的文件，使用 git add 文件名 将该文件添加到暂存区（追踪该文件）

    当文件被修改时，需要重新将其添加至暂存区，也可以使用 git restore 文件名 将工作区内容还原为暂存区的内容

    删除暂存区文件

    使用 git rm --cached 文件名 删除暂存区文件

    注意：，若没写 --cached 会删除工作区里面的文件，慎重操作

    提交到本地库

    使用 git commit -m "日志信息" 文件名，将暂存区的文件提交到本地库，形成历史版本

    查看历史记录

    使用 git reflog 或 git log 查看提交历史记录

    版本穿梭


    使用 git reset --hard 精简版本号 将指针移动到指定版本，将工作区的内容回溯到指定版本

    • 其中，精简版本号可以通过 git reflog 获取

    Git 分支

    分支概述

    分支就是将项目开发中的项目与 release 的项目分离，让开发不会影响到正在运行的版本。

    也可以将每个功能的开发分开，多个功能并行开发然后合并，提高效率。

    分支

    分支操作

    命令 作用
    git branch "分支名" 创建分支（引号可选）
    git branch -m 旧分支名 新分支名 重命名分支
    git branch -d 分支名 删除分支
    git branch [--list] 查看分支, --list 参数可选
    git branch -v 查看分支和目前它指向的版本
    git checkout 分支名 切换到指定分支
    git merge 分支名 将指定分支合并到当前分支

    细说 Git 分支

    删除分支

    在删除分支之前，切换到其他分支，然后使用 git branch -d 分支名 删除分支，删除分支不会产生记录，并且与该分支相关的记录会被删除

    注意：使用该命令后，Git 会检查该分支的文件是否被修改（提交了才算修改）并且已经被合并到其它分支中

    • 若已修改已合并，则马上删除
    • 若未修改未合并，则马上删除
    • 若已修改未合并，则提示 …… not fully merged，若让要删除，使用 git branch -D 分支名 强制删除

    切换到指定分支

    使用 git checkout 分支名 切换到指定分支。但注意：

    • 每次切换分支操作会产生一次记录

    分支合并——正常

    分支合并就是直接将合并的分支的数据直接覆盖到被合并的分支上

    对于单个文件，他在合并之前只在其中一个分支上有改动，则合并不会产生冲突

    分支合并——冲突

    对于单个文件，若其在合并之前在多个分支上有改动，当将两个或以上改动过该文件的分支合并时，会产生冲突

    无冲突
    冲突

    冲突解决：

    冲突发生后，相关文件会以以下格式呈现：

    ...（不冲突的行）

<<<<<<< 发出合并指令的分支名
发出合并分支中对冲突行的更改
=======
被要求合并分支中对冲突行的更改
>>>>>>> 被要求合并过来的分支名

...（不冲突的行）

    要解决冲突，则删除其他符号，且保留一个更改

    提交冲突修改：

    造成冲突后，命令行会变成 (main|MERGING)，使用 git commit -a -m "版本信息" 提交已经解决冲突的文件（注意：不能指定文件名）

    远程库操作——别名

    当我们要访问一个远程仓库的时候，每次都是用这么长的网址属实不方便，为什么不给网址起个名字？

    命令 作用
    git remote [-v] 查看当前已经设定的别名
    git remote add 别名 网址 给网址添加一个别名

    远程库操作——拉取、克隆和推送

    命令 作用
    git push 别名或地址 分支名 推送本地的指定分支到远端
    git pull 别名或地址 分支名 从远端拉取指定分支到本地
    git clone 地址 从远端克隆整个项目到本地
    克隆完成后进入项目文件
    会发现该地址被自动创建了一个别名 origin

    关于它们的区别：

    • push-pull：推送是需要权限的，推送者必须是代码库的协作者
    • pull-clone：两个走不需要权限
      • clone 用于本地没有代码库的时候将整个库拉下来
      • pull 用于更新本地代码，只从远端拉取与本地不同的代码

    Github——（跨）团队协作

    团队协作

    在该账号下 Github 的某个代码库中点击 Settings->Collaborators 即可添加协作者

    跨团队协作

    1. 在 Github 中别人的代码库中点击右上角 Fork 即可将别人的代码库拉取到自己的远程库中

    2. clone 到本地修改完之后，push 到远程库

    3. 在该远程库页面发起 Pull requests

    4. 被请求者会收到请求，处理请求即可

    SSH 免密

    1. 进入 Github，指向头像->Settings->SSH and GBG keys->new SSH key

    2. 公钥可在 系统盘/Users/用户名/.ssh/ 文件夹下找到，若没有该文件夹，使用以下命令创建公钥和私钥

      ssh-keygen -t rsa [-C 任意内容（一些备注之类的，可以写邮箱或者所使用的用户名等）]

    3. 回到任意一个仓库，点击 Code 就可以看到 SSH 链接，使用方法和使用 HTTPS 链接相同，如：

      git push git@github.com:用户名/仓库名 master

    Git in IDEA

    忽略文件——手动配置

    编译器会在项目文件夹内生成与项目无关的文件，这些文件不会影响项目的运行，因此在版本控制时不需要上传这些文件

    配置忽略文件：

    忽略文件的后缀名为 .gitignore，文件名任意

    忽略文件分为 全局的 和 项目的

    • 全局：

      • 忽略文件在任意位置创建，推荐与 .gitconfig 放在一起

      • 创建完之后需要设定指向忽略文件，在 系统盘/Users/用户名/.gitconfig 文件下添加以下语句：

        [core]
    excludesfile = X:/PATH/TO/Java.gitignore

    • 项目：

      • 忽略文件在项目文件夹下，通常 IDEA 已经创建好，看看 .idea/.gitignore

    更多语言的忽略文件见：https://github.com/github/gitignore

    忽略文件——通过插件

    下载插件 .ignore (<-这就是插件名)，在项目名处右键 ->new->.ignore File->.gitignore File，在弹出窗口处勾选对应的语言和编译器（如勾选 Example user template、Java 和 JetBrains）

    注意：.gitignore 文件要和 .git 文件夹在同一目录下，且只对这之后要提交暂存区的文件生效

    • 算了还是用全局的吧……

    将下面的 protected branchs 改为默认主分支的名字（设置 Git 的默认主分支，详见 “Git 的一些问题——初始分支名”）

    添加、提交

    可以通过右键项目名的方式添加到暂存区和提交本地库

    也可以通过上方标签栏的 Git 标签栏进行提交

    切换版本

    通过下面的 Git 分页可以看到版本，右键相应版本，选择 Checkout Revision xxxxxx 即可切换版本

    创建分支、切换分支

    通过菜单添加、切换分支

    通过右下角提示添加、切换分支

    分支合并

    同样是在添加切换分支这个小窗口里面，切换到主分支，选择要合并的另一个分支，选择 merge ...，若无冲突则合并成功，若有冲突则需要进行一些选择

    Git 的一些问题

    初始分支名

    Github 与 IDEA 的初始分支名默认都是 main，但是 Git 默认分支名是master（可能是安装的时候没有设置？），在任意位置使用以下代码更改默认分支名，免去以后重命名分支或合并分支的麻烦：

    git config --global init.defaultBranch main
